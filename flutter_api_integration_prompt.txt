Accountanta Flutter App — API Integration Prompt
=================================================

You are building the Accountanta Flutter mobile application. You MUST integrate exclusively with the Accountanta Finance Tracker REST API documented below. Do NOT invent endpoints, field names, HTTP methods, or response structures. Every network call must match this specification exactly.


BASE URL & VERSIONING
---------------------

All endpoints are prefixed with /api/v1. Use the following base URLs:

  - Production:  https://api.ikondesoft.com
  - Local dev:   http://localhost:3000  or  http://127.0.0.1:3000

All requests and responses use application/json unless the endpoint explicitly uses multipart/form-data.


AUTHENTICATION
--------------

Mechanism: HTTP Bearer JWT

Every protected endpoint requires the header:
  Authorization: Bearer <token>

The JWT token is returned in the data.token field from:
  - POST /api/v1/auth/signup
  - POST /api/v1/auth/login
  - POST /api/v1/auth/google

Store this token securely (e.g., flutter_secure_storage) and attach it to every subsequent request. The token is invalidated server-side on DELETE /api/v1/auth/logout.


RESPONSE ENVELOPE
-----------------

ALL API responses follow one of these two envelope shapes. Never assume a flat response.

Success:
  {
    "status": "success",
    "data": { ... }   // object, array, or primitive
  }

Error:
  {
    "status": "error",
    "error": "Human-readable error message"
  }

For paginated responses, a top-level pagination object is also present:
  {
    "status": "success",
    "data": [ ... ],
    "pagination": {
      "current_page": 1,
      "total_pages": 5,
      "total_count": 98,
      "per_page": 20
    }
  }

Parse data.status and/or HTTP status codes to determine success or failure. Handle every documented HTTP error code (401, 404, 422) explicitly in the UI.

ENDPOINTS
---------

--- AUTHENTICATION ---

POST   /api/v1/auth/signup          None    Create account
POST   /api/v1/auth/login           None    Email/password login
POST   /api/v1/auth/google          None    Google OAuth
DELETE /api/v1/auth/logout          Bearer  Invalidate token
GET    /api/v1/auth/me              Bearer  Current user
POST   /api/v1/auth/forgot_password None    Request OTP
POST   /api/v1/auth/verify_otp      None    Verify OTP → get reset token
POST   /api/v1/auth/reset_password  None    Reset password with token

Signup body (required: email, password, password_confirmation, first_name, last_name; optional: currency):
  {
    "email": "user@example.com",
    "password": "Password123!",
    "password_confirmation": "Password123!",
    "first_name": "John",
    "last_name": "Doe",
    "currency": "USD"
  }
  Response 201: { "status": "success", "data": { "user": {...}, "token": "..." } }

Login body (required: email, password):
  { "email": "user@example.com", "password": "Password123!" }
  Response 200: same shape as signup. Response 401 on bad credentials.

Google OAuth body (required: email, uid; optional: first_name, last_name, photo_url):
  { "email": "user@gmail.com", "uid": "google-uid-string", "first_name": "John", "last_name": "Doe", "photo_url": null }

Forgot password body (required: email). Always returns 200 for security.

Verify OTP body (required: email, otp):
  { "email": "user@example.com", "otp": "123456" }
  Response 200: { "status": "success", "data": { "message": "...", "reset_token": "..." } }
  Response 422 on invalid/expired OTP.

Reset password body (required: token, password, password_confirmation):
  { "token": "<reset_token>", "password": "NewPass!", "password_confirmation": "NewPass!" }

STRICT RULES FOR IMPLEMENTATION
--------------------------------

1.  No invented endpoints. Every API call must match a path in this document exactly.

2.  No field name aliases. Use the exact field names from the spec. Do not rename
    first_name to firstName, transaction_type to type, etc. in your HTTP layer.

3.  Enum values are exact strings.
      - transaction_type:  "income" | "expense" | "transfer"
      - debt status:       "pending" | "paid" | "overdue"
      - category transaction_type: "income" | "expense"

4.  Monetary strings. Amounts come back as strings (e.g., "1000.00").
    Parse with double.tryParse(). Never treat them as raw numbers from JSON.

5.  Date formats. Request bodies that take dates use ISO 8601 ("2026-03-01T00:00:00Z")
    unless the field explicitly uses date format (YYYY-MM-DD), which applies to
    due_date on debts only.

6.  DELETE responses. Successful deletes return 204 No Content with no body.
    Do not attempt to parse a response body.

7.  Pagination. Only GET /api/v1/budgets is paginated server-side.
    Use ?page= and ?per_page= query parameters.
    Read pagination.total_pages to determine if more pages exist.

8.  Token handling. After logout (DELETE /api/v1/auth/logout), clear the stored token.
    On 401, redirect to the login screen and clear stored credentials.

9.  Error handling. Display the error field from error responses to the user.
    Never swallow API errors silently.

10. Content-Type. Set Content-Type: application/json for all JSON requests.
    Use multipart/form-data ONLY for POST /api/v1/profile/upload_photo.
